1. EXECUTIVE SUMMARY & VALUE PROPOSITION
----------------------------------------------------------------------------------------------------
Custom Log-Level Data (LLD) extractions historically represented an operational bottleneck. 
Each custom client data request required an analyst to manually write SQL queries, handle timezone 
conversions, normalize micro-dollar configurations, partition large datasets, and execute bespoke 
delivery protocols (S3, Google Drive, BigQuery, or SFTP). 

The engineered script solves this friction by functioning as a centralized, parametrized data 
orchestration template. In its initial deployment phase, it integrates with Kestra to receive 
structured runtime parameters, completely standardizing the extraction and routing mechanics.

Q2 Business Impact: By shifting from ad-hoc scripting to this scalable template pattern, the data team successfully 
eliminated repetitive coding overhead, saving over 100 manual analyst hours in Q2.

The Next Horizon (End-to-End Autonomy):
While the template currently eliminates manual writing of script logic, an analyst must still manually 
input parameters into Kestra to provision a data workflow. The pipeline is currently being upgraded 
to bridge directly into Freshservice tickets via API. When a client requests an LLD export, the 
Freshservice system parameters will programmatically trigger the Kestra orchestration layer. This 
completely removes human intervention, establishing a true zero-touch, self-service infrastructure.
My goal is to have this completed by end of year, as it stands we are on track.

2. ARCHITECTURAL DATA FLOW & CORE CAPABILITIES
----------------------------------------------------------------------------------------------------
The engine is structured around three programmatic phases: Parameter Parsing, Unified Extraction, 
and Multi-Destination Delivery Routing.


Phase A: Automated Initialization & Token Swapping
- The environment interrogates the platform kernel (`platform.system()`) to map appropriate absolute 
  paths for server configurations (`/home/ec2-user/`).
- It extracts runtime inputs provided by Kestra orchestration environment arrays:
  * Identity context variables: `owner`, `client_name`, `fs_ticket_id`
  * Query context variables: `line_item_ids`, `start_date`, `end_date`, `sample` flags
  * Destination parameters: `final_destination`, `var1`, `var2`

Phase B: Robust Performance Query Ingestion (Redshift)
- Rather than maintaining independent ad-hoc codebases per client, the script leverages a standardized 
  abstract template logic engine. It generates time-window loops using Python's `datetime` and 
  `timedelta` formulas to scan historical table partitions cleanly.
- It translates micro-dollars back to readable decimal formats (`SUM(cost)/1000000.0`), normalizes 
  regional timezones, tracks conversions, and reads millions of log events seamlessly. 
- If the `sample` flag evaluates to `True`, the engine injects a strict `LIMIT 100` boundary clause. 
  This allows analysts or upstream system calls to rapidly validate schema structures in seconds 
  without incurring high database compute costs.

Phase C: Polymorphic Delivery Adapter Matrix
A core highlight of the architecture is its decoupling of data extraction from data routing. 
The script exposes a modular, adaptive interface (`deliver()`) capable of writing compressed in-memory 
data streams across four enterprise-grade endpoints dynamically based on the input payload:

1. GOOGLE_DRIVE  -> Stream-injects binary packets directly via `MediaIoBaseUpload` to specific shared folders.
2. BIGQUERY      -> Provisions schema structures and appends logs cleanly to target Google cloud data tables.
3. S3_BUCKET     -> Ships large-scale compressed objects directly to secure AWS storage buckets.
4. SFTP          -> Establishes programmatic SSH transport tunnels using `paramiko` to deliver data straight to client infrastructure.

3. ADVANCED ENGINEERING HIGHLIGHTS & OPERATIONAL SAFEGUARDS
----------------------------------------------------------------------------------------------------
- In-Memory Buffer Streaming: To protect containerized environments from out-of-memory crashes 
  and disk spatial exhaustion, data outputs are structured utilizing virtual file buffers (`io.StringIO`). 
  The codebase completely avoids intermediate disk writes during dataset processing, processing large data 
  volumes natively in volatile memory buffers.
- Dynamic Slack Notification Layer: The script actively interfaces with WebClient APIs. Upon execution, 
  it maps internal organizational profiles, locates users, translates user names into dynamic Slack 
  Markdown tagging notation (`<@target_slack_id>`), and streams automated alerts directly to internal 
  monitoring channels (`da-lld-notification`). This ensures real-time oversight of system completions or 
  failures.
